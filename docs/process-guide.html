Control Translator

End-to-End Process Guide

Audience: Solution Engineer  ·  Cadence: Once per standard release cycle (~annually)  ·  Version: 1.0 — Current

1. Overview

Control Translator turns a published security standard — such as the New Zealand Information Security Manual (NZISM) — into a deployable Azure custom Regulatory Compliance initiative. It ingests the control catalogue, pulls the current set of Azure built-in policy definitions, uses an AI classifier to map each control to the most appropriate built-in policies, and emits a native Azure initiative (plus Bicep deployment template) ready to assign to your subscription or management group.

The result appears in Defender for Cloud → Regulatory compliance just like a Microsoft-published built-in standard, but is under your control — no dependency on a centrally-managed publishing pathway.

3. One-Time Setup

3.1 Get the code

First time: clone the repository.

PS> git clone <repository-url> control-translator
PS> cd control-translator

Subsequent cycles (annual update): pull the latest version of the tool.

PS> cd control-translator
PS> git pull origin main

⚠ Never overwrite your data/ folder from the repo

The data/ folder contains your working data — mapping decisions (data/mappings/), OOS registers, source CSVs, and the built-in policy cache. These are yours, not the tool's. If you copy or sync the full project folder from a shared location, always exclude data/:

# Correct: sync only code and config, never data
robocopy <source>\src    C:
epos\control-translator\src    /MIR /NFL /NDL
robocopy <source>\config C:
epos\control-translator\config /MIR /NFL /NDL

# Wrong: this overwrites your mapping store and OOS register
# robocopy <source> C:
epos\control-translator /MIR   ← do not use

Overwriting data\mappings zism.json loses all your curated decisions and forces a full re-run. Overwriting data\mappings\global-ignore.json replaces your OOS register with an empty or example file.

3.2 Install

From the control-translator folder in PowerShell:

PS> python -m venv .venv
PS> .venv\Scripts\Activate.ps1        # prompt shows (.venv) when active
PS> pip install -e ".[azure,openai]"   # core + Azure ARM pull + GPT-4o-mini + numpy (needed for embedding retrieval)

⚠ PowerShell activation blocked?

If Activate.ps1 fails with an execution policy error, run this first:
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
Then run .venv\Scripts\Activate.ps1 again.

Also create the folder structure the tool expects:

PS> mkdir data\source   -Force   # where your control framework CSV goes
PS> mkdir data\mappings -Force   # mapping store and OOS register
PS> mkdir data\cache    -Force   # Azure built-in policy cache

3.3 Create your config file

The repository includes a starter config file. Copy it to create your own working copy — this keeps the original clean and lets you maintain multiple configs for different environments or standards.

PS> copy config\nzism-azure.json config\<your-initiative-name>.json

Open config\nzism-myjob.json in VS Code or your editor of choice and update the settings for your environment. The key section to configure is mapping — specifically the classifier and its endpoint.

4. Preparing Your Control Framework

The currently supported input format is a CSV export from your published standard. For NZISM the file has 9 columns:

Columns: Chapter, Section, Sub-Section, Block, Paragraph, Classifications, Compliance, CID, ControlText

ℹ Roadmap note

Website URL and PDF ingestors are on the roadmap. For now, export the standard manually from its publisher's website as CSV. For NZISM: nzism.gcsb.govt.nz.

The data\source folder was created in Section 3.2. Place your CSV there:

# Copy your framework CSV export to the source folder
PS> copy "C:\path\to\your-framework.csv" data\source\your-framework.csv
# NZISM example: copy "...\NZISM-ISM-Document-V3.9.csv" data\source\NZISM-3.9.csv

4.1 Optional — create a chapter subset for testing

For a first run, filtering to a single chapter or section is a fast, cheap way to prove the pipeline before processing all controls. The one-liner below filters by chapter — update the startswith value to match a chapter prefix in your framework. (NZISM example: Chapter 6 has 16 controls out of 1,422 total.)

PS> python -c "
import csv
rows=[r for r in csv.DictReader(open('data/source/NZISM-3.9.csv',encoding='utf-8-sig'))
      if r['Chapter'].startswith('6.') and r['CID'] and r['ControlText']]   # change '6.' to your target chapter
w=csv.DictWriter(open('data/source/nzism-ch6.csv','w',newline='',encoding='utf-8-sig'),fieldnames=rows[0].keys())
w.writeheader(); w.writerows(rows)
print(len(rows), 'controls written')
"

💡 Test run vs full run — which config to use in Section 5

If you created the chapter 6 subset above, update your config's ingest.source to data/source/your-chapter-subset.csv for a quick test. When ready for the full run, change it back to your full CSV path. Everything else in the pipeline is identical — the full run just processes all controls instead of the subset.

4.2 Choose a classification profile (NZISM)

NZISM controls include a Classifications field indicating which security levels the control applies to. For a public-cloud deployment approved only up to a certain level (e.g. NZ Government: up to Restricted), out-of-scope controls can be filtered at ingest so they never reach the LLM — saving time and cost.

Set ingest.classification_profile in your config:

"ingest": {
  "type":                    "nzism",
  "source":                  "data/source/NZISM-3.9.csv",
  "classification_profile":  "restricted"   // all | restricted | protected
}
ProfileIncludesExcludesUse case
all Every control Nothing Full catalogue, no filtering
restricted All Classifications, Restricted/Sensitive, Unclassified Secret, Top Secret, Confidential-only NZ Government Azure (NZISM v3.9: 1,216 of 1,422 controls)
protected As restricted + Protected Above Protected AU Government — IRAP / ISM

💡 Impact on your run

With restricted profile on NZISM v3.9, 206 Secret/Top Secret controls are excluded before any LLM call — reducing the run from 1,422 to 1,216 controls. The excluded controls appear in out-of-scope.json with "source": "classification-filter".

5. Running the Pipeline

5.1 Authenticate

PS> az login --tenant <your-tenant-id>

Replace <your-tenant-id> with your Azure AD tenant ID or domain name (e.g. yourdomain.onmicrosoft.com). Using --tenant avoids credential confusion when you have access to multiple tenants. Find your tenant ID in Azure Portal → Microsoft Entra ID → Overview.

This single login covers the Azure built-in policy catalogue pull and keyless auth for Azure AI Foundry.

5.2 Run

PS> ct run --config config\nzism-myjob.json   # the config you created in Section 3.3

The pipeline does the following on each run:

  1. Pulls Azure built-in policies (~1,000+ definitions) from ARM and caches to data\cache\azure-builtins.json. Skipped on subsequent runs unless cache is deleted.
  2. Ingests all controls from your framework CSV into an OSCAL catalogue. (NZISM v3.9: 1,422 controls across 23 chapters.)
  3. Auto-excludes two categories from the candidate pool before any LLM call: [Preview]: policies (automatic, temporary) and entries in your OOS register file(s) configured under global_ignore (permanent human OOS decisions).
  4. Maps each control — retrieves the top-12 most similar built-in policies (TF-IDF shortlist), then sends each shortlist to the LLM classifier for include/ignore/OOS decisions with rationales.
  5. Builds the initiative artefacts in out\<framework-id>-<version>\. (NZISM example: out\nzism-3.9\.)

💡 Speed up the run — parallel classification

The agentic mapper calls the LLM once per control. By default it runs 5 calls concurrently (configurable via mapping.concurrency). With concurrency: 5 and GPT-4o-mini, a full 1,216-control run takes approximately 35 minutes. Without parallelism it would take ~3 hours.

To adjust: "concurrency": 8 if your rate limits allow; lower to 3 if you see rate-limit errors. Rate-limit errors are logged as ✗ error on <control-id> and retried on the next run.

5.3 Progress output during the run

The pipeline streams progress to the terminal as it works. Each stage announces itself, and the mapping stage prints one line per control — updated in place — so you can see exactly where it is at any moment.

The control line format is: [current/total] [progress bar] pct% control-id decision (confidence · policies). It overwrites in place on each step so long runs don't flood the terminal.

On a subsequent run, carry-forward controls show as carry-forward (include) or carry-forward (ignore) — no LLM call, effectively instant.

5.4 Subsequent runs — the carry-forward

The mapping store (data\mappings\<your-framework>.json) persists decisions across runs. Controls already decided as include or ignore are reused without any LLM call — only undecided controls trigger the classifier. This is why re-runs after the initial pass are fast and cheap.

⚠ When to delete the mapping store

Delete your mapping store file (e.g. data\mappings\nzism.json) when you need decisions to recompute from scratch — for example, after adding a new entry to your OOS register and wanting it to take effect immediately, or when testing a configuration change.

7. Triaging OOS Candidates

After each run, oos-candidates.json lists policies the LLM has flagged as out of scope by default for a shared platform initiative. These are not necessarily wrong policies — they may be perfectly appropriate for an individual adopting organisation to implement as custom policies in their own environment. The distinction is that they require the adopting organisation to supply environment-specific configuration that cannot be set meaningfully in a shared policy set. Including them in a platform standard would produce either incomplete evaluation or misleading compliance posture for every adopting organisation using the initiative.

Common patterns for default OOS:

  • Requires organisation-specific values — the policy needs parameters that vary by environment and cannot be set generically, such as approved OS/SOE image names, admin account names, approved package or namespace lists
  • Requires a pre-deployed agent or extension — evaluation depends on infrastructure (e.g. ChangeTracking agent, Guest Configuration extension) that each adopting organisation must deploy and manage independently
  • Is a process or management control — the policy describes an organisational obligation (e.g. "Develop an incident response plan") that cannot be automatically evaluated by Azure Policy
  • References a technology not universally available — e.g. a protocol version or service capability not yet supported across all services an adopting organisation may use

💡 OOS ≠ non-compliant

Marking a policy as out of scope for this shared initiative does not mean the underlying control is unimportant or unaddressed. Adopting organisations should handle OOS controls through their own custom policies, manual attestation, or compensating controls appropriate to their environment. The out-of-scope.json register provides a documented rationale for each exclusion that can be shared with auditors.

ℹ Two-tier OOS register

Organise your OOS register as two separate files listed under global_ignore in the config:

  • global-ignore.json — excluded from all frameworks (process controls, In-Guest agent, Manual-effect policies). Shared across NZISM, IRAP, ISO 27001, etc.
  • <standard>-ignore.json — excluded from this standard only. A policy here can still appear in another standard's initiative.
"global_ignore": [
  "data/mappings/global-ignore.json",
  "data/mappings/nzism-ignore.json"
]

💡 Note on [Preview]: policies

[Preview]: policies are automatically excluded at engine time and do NOT appear in oos-candidates.json. They are tracked in out-of-scope.json under "source": "auto-preview" and will be flagged for reconsideration when they go GA. You do not need to add them to your OOS register manually.

Process

  1. Open out\<your-output-folder>\oos-candidates.json
  2. Review each entry — check display_name and oos_reason
  3. For entries you agree with: decide which register it belongs in, copy the entry to that file, add oos_date (today's date), and remove the first_seen_control field
  4. For entries you disagree with: leave them — they are suggestions only and will be re-proposed next run
  5. After updating an OOS register file: delete the mapping store, then re-run

OOS register entry format

{
  "policy_id": "/providers/Microsoft.Authorization/policyDefinitions/221aac80-...",
  "display_name": "[Preview]: ChangeTracking extension — Windows VM",
  "reason": "Requires ChangeTracking agent pre-deployed on each VM — not generically enforceable across agencies.",
  "oos_date": "2025-09-03"
}

ℹ Policy ID format

You can use either the full ARM resource path (/providers/Microsoft.Authorization/...) or just the bare GUID — the tool normalises both automatically.

8. Handling OOS Reconsidered

The file oos-reconsidered.json only appears when the system detects that a previously excluded policy has changed state. Two triggers:

TriggerWhat happenedAction
Preview → GA A policy that was [Preview]: when OOS'd now has a GA display name in the catalogue — Microsoft shipped it Remove from the relevant OOS register file. Delete mapping store. Re-run — it will now be proposed fresh.
Policy removed A policy in your OOS register no longer exists in the Azure built-in catalogue — deprecated or replaced Remove from the relevant OOS register file (it's irrelevant). Optionally note in project records if it had a replacement.

💡 Why this matters annually

Microsoft ships new GA policies regularly. Policies that were legitimately preview-excluded twelve months ago may now be production-ready and appropriate to include. The staleness check runs automatically every time and ensures the OOS register doesn't become stale cruft.

9. Working the Pending Review Queue

Pending review items are controls where the LLM proposed one or more built-in policy mappings but either (a) the confidence score was below the auto-approve threshold, or (b) auto_approve is set to false in the config. These require a human decision requiring sign-off from your standard's governing authority.

9.1 View proposals

PS> ct review --config config\<your-config>.json

Each pending item shows:

[0.82] 06.2.5.C.01: A vulnerability assessment solution should be enabled on your virtual machines
         keyword baseline; matched on: vulnerability, assessment

Format: [confidence] control-id: proposed-policy-name followed by the rationale.

9.2 Authority sign-off step

Before finalising, share the pending proposals with the relevant authority for your standard — the body responsible for publishing and maintaining the framework (e.g. the national cyber security centre, a certification body, or your organisation's CISO). They review whether each control→policy mapping accurately represents the standard's intent.

Share the ct review output and the rationales from the mapping store. Incorporate their feedback by editing the mapping store before the next run.

9.2b Excel export / import (recommended for authority sign-off)

Rather than hand-editing the JSON store, export the entire review queue to Excel, share it with the authority (e.g. GCSB/NCSC), and import their decisions back in one step.

PS> pip install openpyxl                    # one-time
PS> ct export-review --config config\<your-config>.json --output review.xlsx

The workbook has three sheets:

  • Pending Review — control text, proposed policy, confidence, LLM rationale, and a yellow Decision column with a dropdown: Include / Ignore / Skip
  • OOS Candidates — policies the LLM flagged, with dropdown: Add to nzism-ignore / Global ignore / Skip
  • Preview Excluded — informational list of auto-filtered [Preview]: policies

Share review.xlsx with the authority. They fill in the Decision columns and return the file. Then import:

PS> ct import-review --config config\<your-config>.json --input review.xlsx
PS> ct run --config config\<your-config>.json   # rebuild with approved decisions

💡 OOS candidates in the workbook

The OOS Candidates sheet lets the authority mark policies for nzism-ignore.json (NZISM-specific) or global-ignore.json (all standards). Import writes them directly — no manual JSON editing required.

9.3 Approve or reject in the mapping store

Open your mapping store file (e.g. data\mappings\nzism.json). Find the control by its id and change the decision field:

// Approve a mapping (policy correctly covers this control)
"decision": "review"  →  "decision": "include"

// Reject a mapping (no appropriate built-in policy found)
"decision": "review"  →  "decision": "ignore"

Save the file, then re-run to rebuild the initiative with the updated decisions:

PS> ct run --config config\<your-config>.json

Approved controls now appear in policySet.json. Ignored controls are excluded.

9.4 First-pass strategy — use auto_approve

For an initial first-cut output, set auto_approve: true and confidence_threshold: 0.7 in the config. This auto-approves high-confidence proposals so you get a populated initiative immediately. Review the auto-approved set as a batch, then switch back to auto_approve: false for the final authority sign-off pass.

💡 Batch vs individual review

The mapping store is a plain JSON file. For bulk approvals, find all "decision": "review" entries with confidence above your threshold and change them in one edit — no need to process one at a time.

ℹ Two independent version numbers

The initiative has two completely separate version numbers that should never be coupled:

  • build.initiative_version (e.g. 1.0.0, 1.1.0) — your deployment semver. Bump it whenever you change the policy set: add a control, remove an OOS policy, update a parameter. This is what Azure tracks for deployment management.
  • framework.version (e.g. 3.9) — the standard document version. Appears only in the initiative description text (NZISM v3.9. ...). Updating the standard to v4.0 does not automatically change the initiative version — that is your decision, based on what policy set changes you make in response.

Example: NZISM publishes v4.0. You update framework.version: "4.0" and, after reviewing the delta, decide to add 12 new controls and remove 3 OOS entries. You bump initiative_version from 1.2.0 to 1.3.0. The description now reads "NZISM v4.0. ..." and Azure sees a v1.3.0 initiative — the two versions reflect two different things.

11. The Annual Update Cycle

When the standard authority publishes a new version:

  1. Export the new CSV from the updated standard and place it in data\source\. Update ingest.source in your config to point to the new file.
  2. Update framework.version in the config (e.g. "3.9" → "4.0"). This sets the ISM version string in the initiative description.
  3. Bump build.initiative_version semver (e.g. "1.0.0" → "1.1.0"). This is independent of the standard's document version — it tracks your policy set changes.
  4. Do NOT delete the mapping store. Carry-forward means only truly new controls need fresh LLM calls. Controls whose text has changed will be re-proposed; unchanged controls reuse prior decisions. A typical annual delta is 30–90 new or modified controls.
  5. Run ct review and work through the three queues: OOS Reconsidered first, then OOS Candidates, then Pending Review.
  6. Re-deploy with the new initiative version.

⚠ Back up the mapping store

data\mappings\<your-framework>.json is your year-over-year institutional knowledge — the curated set of human-approved control→policy decisions. Check it into source control (Git). Losing it means re-running all LLM calls from scratch and repeating the authority sign-off cycle.

13. Troubleshooting

13.1 Content filter / jailbreak detection (Azure AI Foundry)

When using Azure AI Foundry as the classifier, Azure OpenAI's content safety system may block a request with a 400 content_filter error. The error message will include "jailbreak": {"detected": true, "filtered": true}.

Why it happens: Azure's jailbreak detector monitors for prompt patterns that resemble attempts to override model safety instructions. Security standards like NZISM contain control text covering topics such as penetration testing, vulnerability exploitation, malicious code, and classified information handling. Combined with the classifier's structured instructions, this can trigger a false positive.

What the tool does automatically: the classifier catches the error, prints a warning, and falls back to the heuristic (token-overlap) classifier for that specific control. The run continues uninterrupted. You will see:

   ⚠  content filter: 16.1.27.C.02 — using heuristic fallback

The heuristic result for that control is saved to the mapping store like any other result and the run continues to the next control.

💡 To retry a content-filtered control with the LLM

Open your mapping store JSON file, find the entry for the flagged control id, and delete that entry. On the next ct run, the engine will re-attempt that control with the LLM. If it triggers the filter again, the heuristic fallback will apply again — consider reviewing and manually setting the decision for controls that consistently trigger the filter.

ℹ If your run crashed before the fix

If a previous run crashed with a content filter error before the automatic fallback was in place, your mapping store still contains all decisions made up to the crash point. Simply re-run — the carry-forward mechanism will skip decided controls and the classifier will only re-attempt the ones not yet decided. No work is lost.

13.2 Slow runs — improving throughput

The agentic mapper calls the LLM once per control sequentially. For a full 1,422-control run, this typically takes 8–20 minutes depending on API rate limits. Strategies to reduce time:

  • Run a chapter subset first — test with Chapter 6 (16 controls, Section 4.1) to validate classifier quality before committing to the full run.
  • Use engine: keyword for a free first cut — produces a baseline initiative instantly. Switch to engine: agentic and auto_approve: false for the quality pass where the LLM only processes controls not yet decided.
  • Re-runs are near-free — carry-forward means only new or undecided controls trigger LLM calls. After the first full run, subsequent annual updates typically involve 30–90 new controls, not 1,422.

💡 Run interrupted? Just re-run

If a run is interrupted (Ctrl+C, crash, content filter before fallback was added), the mapping store has everything decided so far. Re-running picks up from where it stopped — only undecided controls trigger new LLM calls.
